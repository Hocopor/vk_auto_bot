## Журнал решений

- 2026-06-25 — **Стек Python** (а не Node): решающий фактор — OCR чеков на русском
  (Tesseract/pytesseract — родная экосистема), зрелые ВК-либы, быстрая сборка
  админки на FastAPI+Jinja2.
- 2026-06-25 — **PostgreSQL** как единственный источник истины (по требованию
  разработчика); Google Sheets — только публичное зеркало.
- 2026-06-25 — **Проверка оплаты без банка**: статичный QR не позволяет надёжно
  сматчить платёж с человеком. Делаем OCR-распознавание суммы/реквизитов + тумблер
  автоподтверждения + ручную модерацию в админке (статусы меняются в обе стороны).
- 2026-06-25 — **Кол-во постеров = floor(сумма ÷ цена)**; несоответствия → ручная
  очередь.
- 2026-06-25 — **Google-таблица создаётся автоматически** при создании мероприятия
  и привязывается к нему.
- 2026-06-25 — **OCR — Tesseract локально** (без облака): бесплатно, перс. данные
  не уходят наружу; цена — ниже точность на плохих фото (больше уйдёт в ручную
  очередь, что приемлемо).
- 2026-06-25 — **Админка по IP** (заказчику без домена), демо у разработчика — по
  своему домену; HTTPS опционально.
- 2026-06-25 — **Скрины из TZ — это конкурент**: копируем механику, не данные.
  «СИГМА» не используется; получатель платежа у заказчика свой, настраивается.
- 2026-06-25 — **Reverse proxy: Caddy вместо nginx** (по требованию заказчика). На
  боевом сервере уже работает Caddy с кучей других сайтов — чтобы их не задеть,
  админка подключается отдельным фрагментом (`caddy/vk_admin.caddy`) через
  `import /etc/caddy/conf.d/*.caddy` на отдельном порту 8080. nginx-конфиг удалён.
- 2026-06-25 — **Автодеплой-скрипт удалён** (`scripts/deploy.sh`). Заказчик хочет
  деплоить руками — в README раздел «Обновление на сервере» переписан в копипаст-
  инструкцию по шагам (git pull → pip → alembic → reload caddy → restart services).
- 2026-06-25 — **Всё параметризуется в админке** на каждое мероприятие: цена билета,
  кол-во билетов, **кол-во победителей**, **сроки работы**. Ничего не захардкожено.
- 2026-06-25 — **Один админ из `.env`** (логин открыто, пароль bcrypt-хэш); таблицы
  admins нет.
- 2026-06-25 — **Удаление мероприятия — каскадное**, чистит только его данные
  (ON DELETE CASCADE по event_id) + Google-лист; остальные не трогаются. Остановка =
  бот не реагирует.
- 2026-06-25 — **Собираем только имя** (без ФИО) + телефон. Оплата без имени/телефона
  НЕ блокирует присвоение номеров — выдаём + напоминаем участнику. Вне сценария бот
  молчит.
- 2026-06-25 — **Отзыв номеров через админку** (статус `revoked`): для непрошедшей
  оплаты/поддельных чеков — номера освобождаются и снова доступны, уходят из таблицы.
- 2026-06-25 — **Все тексты бота настраиваются на мероприятие**, не захардкожены:
  при создании предзаполняются дефолтными шаблонами, потом редактируются/заменяются.
- 2026-06-25 — **Фаза 1 завершена**: архитектура и структура репозитория — в
  `ARCHITECTURE.md`; воркер = polling 5 сек; номера — `generate_series ORDER BY
  random()`; плейсхолдеры — безопасный рендер (не сырой `str.format`).
- 2026-06-25 — **Этап 2.1 (каркас) принят**: структура пакетов, requirements,
  config (pydantic-settings), db (async engine/Base/get_session), Alembic async
  (env.py берёт URL из settings, target_metadata=Base.metadata).
- 2026-06-25 — **`participants` — per-event** (развилка §6): добавлено `event_id`
  (FK ON DELETE CASCADE), unique `(event_id, vk_user_id)`. ТЗ давало эту
  уникальность и требование «удаление чистит только данные события»; один человек
  в N розыгрышах = N строк, имя/телефон/номера изолированы по событию.
- 2026-06-25 — **`assign_unique` на Python** (а не PG `generate_series ORDER BY
  random()` из ARCHITECTURE §4.1): загрузить занятые → `random.sample` по свободным.
  Причина: диапазоны билетов малы (сотни/тысячи), вариант переносим (PG/SQLite) и
  тестируется без живого Postgres; уникальный индекс по-прежнему страхует от гонок.
  ARCHITECTURE §4.1 обновлён.
- 2026-06-25 — **Победители — display-only** (этап 2.11): таблицы winners в схеме §6
  нет, генератор показывает результат розыгрыша без персиста. Повторный розыгрыш = новый
  случайный выбор. Соответствует ТЗ.
- 2026-06-25 — **Пин `bcrypt>=4.0,<4.1`**: passlib 1.7.4 несовместим с bcrypt>=4.1.
  Не снимать верхнюю границу, пока passlib не заменён.
- 2026-06-25 — **ФАЗА 2 ЗАВЕРШЕНА**: все этапы 2.1–2.13 приняты. 105 тестов зелёных,
  1 skip (реальный OCR без бинаря Tesseract). Серверная часть (бот+воркер) и веб-
  админка (auth, мероприятия, модерация, участники, победители) готовы; артефакты
  деплоя (systemd/Caddy/README, ручной деплой) готовы. Остаётся боевое развёртывание
  заказчиком (VK-токен, Google SA-json, сервер по IP) + ручной прогон сценария в ВК.

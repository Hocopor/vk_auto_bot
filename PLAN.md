## Фаза 9 — Контроль лимита номеров (overselling) + остановка бота по исчерпанию

> Делать ПОСЛЕ текущего плана Фазы 8 (заказчик попросил поместить в конец). Сильно
> пересекается с багом статусной машины модерации (память
> `bug-moderation-status-churn-no-delivery`) и этапами 8.3/8.5 — присвоение/освобождение
> номеров лучше рефакторить одним заходом. Оркестратор проектирует, перед кодом —
> уточнить развилки у заказчика (помечены ⚑).

**ПРОБЛЕМА (как сейчас, по коду):**
- Лимит билетов = диапазон `[number_min..number_max]`. Сейчас он НЕ ограничивает приём:
  бот реагирует на кодовое слово и принимает чеки, пока событие `is_active` и в сроках
  (`dialog.find_matching_event` → `is_event_open`), независимо от остатка свободных номеров.
- При одобрении покупки, которой нужно больше номеров, чем свободно, `numbers.assign_unique`
  (всё-или-ничего, `numbers.py:57`) бросает `NumbersExhausted`; воркер (`worker.py:73-80`)
  только пишет в лог `... — ADMIN ALERT` и `continue`. Т.к. `numbers_assigned` остаётся
  False, воркер ретраит ту же покупку КАЖДЫЕ 5с ВЕЧНО (наблюдали на боевом: event 7,
  `requested 6, available 5` — спам в журнале). Участник номеров не получает; в админке
  НИЧЕГО не видно — алерт только в серверном логе.
- Итог: заданный заказчиком лимит по факту превышается («продаём» больше билетов, чем есть),
  и об этом в админке никто не узнаёт. Лимит бесполезен.

**ТРЕБОВАНИЯ ЗАКАЗЧИКА (3 шт.):**
- **A.** Не превышать лимит номеров (не «продавать» больше, чем есть в диапазоне).
- **B.** Когда номера кончились, а оплат больше, чем номеров — ЗАМЕТНО пометить это админу
  В UI (а не только в логах).
- **C.** По исчерпании номеров бот ПЕРЕСТАЁТ реагировать на кодовое слово (новые участники
  не могут начать оплату).

- [ ] **9.1 Модель ёмкости.** `capacity = number_max - number_min + 1`;
      `assigned = COUNT(PosterNumber WHERE event_id)`; `free = capacity - assigned` (уже есть
      `numbers.free_count()`). Хелпер `numbers.event_capacity(session, event_id)` →
      `{capacity, assigned, free}`. ⚑ Развилка «прогнозной» ёмкости: учитывать ли
      забронированные-но-не-присвоенные (покупки pending_ocr/manual_review/approved-not-assigned)?
      Предложение к утверждению: бота закрывать по ПРОГНОЗНОЙ ёмкости
      `free_proj = capacity - assigned - (число открытых неотклонённых покупок без номеров,
      минимум 1 номер на каждую)` — иначе между «оплатил» и «одобрено» наберётся оплат больше
      ёмкости. Точную модель (минимум 1 vs floor по OCR-сумме) утвердить с заказчиком.
- [ ] **9.2 Требование C — гейтинг бота по ёмкости.** Бот не реагирует на кодовое слово,
      когда `free<=0` (или прогнозный free<=0). Реализация: т.к. `is_event_open` — чистая
      синхронная без БД, добавить async-хелпер `is_event_accepting(session, event, now)` =
      `is_event_open(...) AND free(_proj)>0` и звать его в `find_matching_event` после
      `is_keyword_match`. ⚑ Сообщение участнику при исчерпании: по умолчанию — МОЛЧАТЬ
      (как вне сценария, заказчик сказал «бот не реагировал»); опционально тумблер+текст
      «все билеты разобраны» (записать как опцию, по умолчанию выкл). Чек от юзера, который
      УЖЕ в диалоге (получил инструкцию ДО исчерпания) и шлёт чек после — `resolve_event_for_receipt`
      ПРИНЯТЬ (не терять оплату!), он попадёт в shortfall при модерации (см. 9.4), не глотать.
- [ ] **9.3 Требование A — не оверселлить при одобрении.** Перед переводом покупки в approved
      (ручной И авто-апрув 8.3) проверять остаток: если `posters_count > free`. ⚑ Развилка:
      (а) БЛОКИРОВАТЬ approve с явным сообщением админу («Свободно N, требуется M — освободите
      номера/расширьте диапазон/уменьшите кол-во») — рекомендация по умолчанию (заказчик хочет
      «не превышать лимит»); (б) ЧАСТИЧНОЕ присвоение N доступных + пометка недостачи (сейчас
      `assign_unique` всё-или-ничего — пришлось бы доработать). Утвердить вариант. Авто-апрув
      при нехватке → НЕ авто-approved, а в ручную с флагом shortfall/needs_attention.
- [ ] **9.4 Требование B — заметная пометка + конец лог-спама.** Колонка `Purchase.numbers_shortfall`
      (Int nullable — сколько номеров не хватило; миграция). Воркер при `NumbersExhausted` вместо
      «лог+вечный continue» → проставляет `numbers_shortfall = requested-available` и ЛАТЧ (не
      ретраит каждый тик, пока ёмкость не изменилась). UI: громкий красный бейдж в карточке
      покупки и в канбане/списке модерации (увязать с `needs_attention` из 8.9.1):
      «⚠ НЕ ХВАТИЛО НОМЕРОВ: нужно M, свободно N». Событие-уровень: в списке мероприятий и в
      форме — индикатор «Номера: assigned/capacity (свободно N)» + бейдж «ИСЧЕРПАНО — бот не
      принимает новых» при free<=0. Убрать спам в логах каждые 5с.
- [ ] **9.5 Восстановление ёмкости.** При расширении `number_max` (валидация: нельзя сузить
      ниже максимального уже присвоенного номера) и при освобождении номеров (reject/revoke,
      `free_numbers`) → сбрасывать `numbers_shortfall`/латч у затронутого события, чтобы воркер
      ПЕРЕОБРАБОТАЛ shortfall-покупки и выдал номера, а бот снова «открылся».
- [ ] **9.6 Тесты (КП).** (1) бот молчит на кодовое слово при free<=0 (юнит на
      `find_matching_event`/`is_event_accepting`); (2) approve при нехватке → блок/частично с
      ВИДИМЫМ флагом, без тихого вечного ретрая; (3) воркер при `NumbersExhausted` ставит
      shortfall и НЕ ретраит каждый тик; (4) освобождение номеров/расширение диапазона →
      бот снова принимает, shortfall переобрабатывается и номера выдаются; (5) инвариант:
      сумма присвоенных номеров НИКОГДА > capacity. **Скилы:** test-driven-development,

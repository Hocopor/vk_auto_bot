# ТЗ — ВК-бот «Розыгрыш постеров» + админка

> Итоговое техническое задание. Сформировано из переписки с заказчиком (`TZ.md`)
> и согласований с разработчиком. Источник «сырого» ТЗ — `TZ.md`.
> Статус развилок: **все ключевые закрыты** (см. «Принятые решения»). Открытые
> вопросы к заказчику — в конце документа (не блокируют старт разработки).

---

## 1. Краткое описание

Сервис из двух частей:

1. **ВК-бот** в сообществе заказчика. Реагирует на кодовое слово, выдаёт инструкцию
   по оплате с QR-кодом, принимает чек, проверяет оплату (OCR + ручная модерация),
   присваивает участнику случайные «номера постеров» и фиксирует его в публичной
   Google-таблице.
2. **Веб-админка**. Создание и настройка «мероприятий» (розыгрышей), модерация
   чеков, просмотр участников, генератор победителя.

Бизнес-модель: человек покупает N постеров → получает N случайных номеров из
заданного диапазона → по окончании продаж среди всех номеров выбирается победитель.

---

## 2. Глоссарий

- **Мероприятие (event)** — отдельный розыгрыш: кодовое слово, диапазон номеров,
  цена постера, шаблоны сообщений, QR-код, привязанная Google-таблица.
- **Участник (participant)** — пользователь ВК, написавший боту.
- **Чек (receipt)** — изображение/документ оплаты, присланный участником.
- **Покупка (purchase)** — факт оплаты: чек + сумма + кол-во постеров + статус.
- **Номер постера (poster number)** — уникальное в рамках мероприятия случайное
  целое число из диапазона, присвоенное участнику за одну единицу покупки.

---

## 3. Роли

- **Участник** — взаимодействует только с ботом в ВК.
- **Админ** — **один аккаунт** (логин/хэш пароля из `.env`). Работает в веб-админке:
  создаёт/останавливает/удаляет мероприятия, модерирует чеки, смотрит участников,
  запускает розыгрыш. (Аутентификация — см. §9.)

## 4. Принятые решения (согласовано)

| Тема | Решение |
|------|---------|
| Стек | Python 3.12, бот на `vkbottle` (или `vk_api`), админка FastAPI + Jinja2, ORM SQLAlchemy 2.x (async) + asyncpg |

| Кол-во постеров | `floor(сумма_оплаты / цена_постера)`; остаток/несоответствие → в ручную очередь |
| Проверка оплаты | OCR (Tesseract, локально) + тумблер автоподтверждения на уровне мероприятия |
| OCR-движок | Tesseract (rus+eng), локально на сервере, без облака |
| Хостинг | cloud.ru, Linux, доступ к админке заказчика — **по IP** (без домена); демо разработчика — по своему домену |
| Инфраструктура | ВК-сообщество и токен — есть; Google-аккаунт для Sheets — есть |
| Данные от участника | Участник присылает **чек + имя + номер телефона** (по инструкции в сообщении). Имя — без полного ФИО. Всё это собираем и храним |
| Сверка OCR | Ожидаемый получатель платежа задаётся в мероприятии (организация заказчика — НЕ «СИГМА», это из примера конкурента) + сумма. По этим признакам работает автоподтверждение |
| Настройки мероприятия | Через админку, отдельно на каждое мероприятие: кодовое слово, цена билета, кол-во билетов (диапазон номеров), **кол-во победителей**, **сроки работы (начало/конец)**, шаблоны сообщений, QR, получатель платежа, тумблер автоподтверждения |
| Управление мероприятиями | Несколько одновременных мероприятий с разными кодовыми словами/настройками; остановка (бот перестаёт реагировать); **удаление вместе со всеми связанными данными в БД** — только данные этого мероприятия, остальные не трогаются |
| Админ | **1 аккаунт.** Логин и хэш пароля берутся из `.env` (логин — открыто, пароль — хэш). Отдельной таблицы админов нет |

### Пример из скринов — ЧУЖОЙ функционал (копируем механику, не данные)

Скрины (`tz/photo_*.jpg`) — это **конкурент**, мы копируем его функционал, но НЕ
данные. Конкретные значения у заказчика будут свои и настраиваются в админке.
В качестве иллюстрации механики (значения примерные, у заказчика другие):

- Кодовое слово (пример): «Акула»
- Цена билета: настраивается (у конкурента было 999 ₽; 2 билета — 1998₽ и т.д.)
- Кол-во билетов (диапазон номеров): настраивается (у конкурента было до 13999)
- Получатель платежа для сверки OCR: **организация заказчика** (НЕ «СИГМА»)
- 1-е сообщение: промо + картинка с QR (персональное обращение `{name}`)
- 2-е сообщение: подтверждение с присвоенными номерами + ссылка на таблицу

## 5. Архитектура

Один сервер (cloud.ru), несколько процессов под управлением systemd:

```
┌─────────────────────────────────────────────────────────┐
│ cloud.ru (Linux, Ubuntu)                                  │
│                                                           │
│  ┌────────────┐   ┌──────────────┐   ┌────────────────┐  │
│  │ vk-bot     │   │ admin-web    │   │ PostgreSQL     │  │
│  │ (long poll)│──▶│ (FastAPI)    │──▶│                │  │
│  │            │   │ Jinja2 UI    │   │                │  │
│  └─────┬──────┘   └──────┬───────┘   └────────────────┘  │
│        │                 │                                │
│        ▼                 ▼                                │
│  ┌────────────┐   ┌──────────────┐                       │
│  │ OCR module │   │ Sheets sync  │──▶ Google Sheets API   │
│  │ (Tesseract)│   │ (gspread)    │                       │
│  └────────────┘   └──────────────┘                       │
│                                                           │
│  Caddy (reverse proxy, :8080 → admin-web)                 │
└───────────────────────────────────────────────────────────┘
```

Компоненты (модули внутри одного репозитория):

- **`bot/`** — воркер ВК: Long Poll, обработка сообщений, FSM-логика диалога.
- **`admin/`** — FastAPI-приложение: маршруты, Jinja2-шаблоны, аутентификация.
- **`core/`** — общий слой: модели SQLAlchemy, бизнес-логика (присвоение номеров,
  расчёт кол-ва, статусы), сервисы.
- **`ocr/`** — обёртка над Tesseract: распознавание + парсинг суммы/реквизитов.
- **`sheets/`** — интеграция с Google Sheets (создание, синхронизация).
- **`shared config`** — `.env` (токены, креды), общий для бота и админки.

Бот и админка — отдельные процессы, общаются **только через БД** (бот пишет
покупки/чеки, читает статусы и шаблоны; админка меняет статусы; смена статуса
на «подтверждён» триггерит присвоение номеров и отправку второго сообщения).

> Триггер на «подтверждён» реализуется так: при подтверждении в админке покупка
> получает статус `approved` и флаг `numbers_assigned=false`. Воркер бота
> периодически (раз в N секунд) или через NOTIFY/LISTEN Postgres подхватывает такие
> покупки, присваивает номера, шлёт второе сообщение, ставит `numbers_assigned=true`.
> (Реализация механизма уведомления — на этапе проектирования модуля; по умолчанию
> polling раз в 5 сек как простой надёжный вариант.)

## 7. Логика бота (потоки)

### 7.1 Кодовое слово → инструкция
1. Участник пишет сообщение. Бот нормализует текст, ищет среди **активных**
   мероприятий совпадение `keyword`. Мероприятие активно, если `is_active=true`
   **и** текущее время в пределах `starts_at`…`ends_at` (если заданы).
2. Если найдено: бот сохраняет/обновляет `participant`, запоминает контекст
   (ждём чек по этому event), отправляет `msg_instruction` + вложение QR-кода.
3. Если активных мероприятий с таким словом нет — **бот молчит** (никакой реакции
   на сообщения вне сценария).
4. Поддерживается несколько активных мероприятий одновременно — у каждого своё
   кодовое слово, свои настройки и шаблоны.

### 7.2 Приём чека + имени + телефона
По инструкции в 1-м сообщении участник присылает **чек (фото/документ), имя и
номер телефона**. Это может прийти одним или несколькими сообщениями.

1. Участник присылает изображение/документ (фото, скрин, pdf-вложение) и текст.
2. Бот определяет, к какому мероприятию относится (по контексту диалога; если
   контекста нет — к последнему активному, по которому участник писал кодовое
   слово; иначе **молчит**, ждёт кодовое слово).
3. **Сбор имени и телефона:**
   - Телефон извлекается из текста регэкспом (российские форматы: `+7`, `8`,
     `7`, с разделителями и без).
   - Имя — оставшийся осмысленный текст сообщения (только имя, без полного ФИО;
     или явный запрос ботом, если не распознано).
   - Имя/телефон можно дослать позже; оба правятся админом в админке.
4. Сохраняет файл чека, считает `receipt_hash`. **Анти-дубль:** если такой хэш
   уже есть — помечает в ручную очередь с пометкой «возможный дубль».
5. Создаёт `purchase` со статусом `pending_ocr`, запускает OCR (§8).
6. Отвечает участнику: «Чек принят, проверяем» (текст настраивается).

### 7.3 Подтверждение → присвоение номеров → 2-е сообщение
1. Покупка получила статус `approved` (авто или вручную).
2. Воркер присваивает `posters_count` уникальных случайных номеров из диапазона
   мероприятия (§8.3). **Присвоение НЕ блокируется отсутствием имени/телефона** —
   номера выдаются, занятые номера не достаются другим участникам.
3. Отправляет участнику `msg_after_payment` (с подстановкой присвоенных номеров,
   ссылки на таблицу — через плейсхолдеры, см. §10).
4. **Если у участника не указано имя и/или телефон** — дополнительно отправляет
   напоминание прислать недостающие данные (текст настраивается).
5. Обновляет БД и зеркалит запись в Google-таблицу.

### 7.4 Шаблоны сообщений — настраиваемые тексты + плейсхолдеры
**Все тексты бота настраиваются на каждое мероприятие через админку и НЕ
захардкожены.** При создании мероприятия каждое поле заполняется дефолтным
шаблоном; админ может его отредактировать, либо стереть и ввести свой.
Настраиваемые тексты: `msg_instruction`, `msg_after_payment`,
`msg_receipt_received`, `msg_need_contacts`.

В текстах поддерживаются подстановки, например: `{name}`, `{numbers}`,
`{sheet_url}`, `{count}`, `{price}`. Точный список фиксируется при реализации и
выводится подсказкой в редакторе. Дефолтные шаблоны хранятся в коде как
константы-болванки и копируются в поля мероприятия при создании (после этого
живут в БД и редактируются независимо).

## 8. OCR и автоподтверждение

### 8.1 Распознавание
- Движок: **Tesseract** (`pytesseract`), языки `rus+eng`.
- Препроцессинг изображения (через `Pillow`/`opencv`): перевод в ч/б, повышение
  контраста, ресайз — для роста точности.
- Из `ocr_raw_text` регэкспами извлекается **сумма** (приоритет — крупнейшее
  денежное значение / по ключевым словам «Итого», «Сумма», «Перевод», «₽»).
- **Сверка получателя:** в тексте чека ищется `expected_recipient` мероприятия
  (организация заказчика, задаётся в админке; сравнение нечёткое, без учёта
  регистра/кавычек). Это основной сигнал валидности платежа вместе с суммой.

### 8.2 Логика статусов
- `pending_ocr` → запускается OCR.
- Если `auto_confirm = true` у мероприятия **и** OCR уверенно распознал сумму
  `ocr_amount`, кратную/достаточную относительно `price`, **и** в чеке найден
  `expected_recipient` (получатель — организация заказчика) → статус
  `auto_confirmed` → далее `approved`.
- Иначе → `manual_review` (отдельная колонка/очередь в админке).
- Если `auto_confirm = false` → всегда `manual_review` (OCR только подсказывает).
- Админ может вручную перевести покупку в `approved` или `rejected` **в обе
  стороны** в любой момент.

### 8.3 Расчёт кол-ва и присвоение номеров
- `posters_count = floor(amount / price)`.
- Если `amount` не делится нацело или меньше `price` → ручная очередь с пометкой
  (админ решает: одобрить с округлением, запросить доплату, отклонить).
- Номера: `posters_count` штук, **уникальных** в рамках мероприятия, случайных
  из `[number_min, number_max]`. Реализация без коллизий: выборка случайных
  значений с проверкой по уникальному индексу `(event_id, number)` в транзакции;
  при исчерпании диапазона — алерт админу, присвоение приостанавливается.

## 9. Веб-админка (FastAPI + Jinja2)

### 9.1 Аутентификация
- **Один аккаунт.** Логин и хэш пароля из `.env` (`ADMIN_LOGIN` открыто,
  `ADMIN_PASSWORD_HASH` — bcrypt-хэш). Проверка пароля сверкой с хэшем. Сессия
  через cookie.
- Доступ к админке заказчика — по IP сервера (без домена). HTTPS — опционально
  (self-signed/по домену у разработчика для демо).

### 9.2 Раздел «Мероприятия»
- Список мероприятий, статус (активно/остановлено), сроки.
- Создание/редактирование, все настройки на каждое мероприятие отдельно:
  название, кодовое слово, **цена билета**, **кол-во билетов** (диапазон номеров
  min/max), **кол-во победителей**, **сроки работы (начало/конец)**, редакторы
  всех текстов бота (1-е сообщение, 2-е сообщение, «чек принят», напоминание о
  контактах), загрузка QR-картинки, тумблер автоподтверждения, получатель платежа
  (организация заказчика для OCR-сверки).
- **Тексты при создании предзаполняются дефолтными шаблонами** (§7.4); админ может
  отредактировать или полностью заменить любой из них. Ничего не захардкожено.
- **При создании** — автоматически создаётся Google-таблица и привязывается
  (`sheet_id`), ссылка показывается админу.
- **Остановка** (деактивация): бот перестаёт реагировать на кодовое слово
  мероприятия. Включение обратно — там же.
- **Удаление мероприятия** — удаляет ВСЕ связанные данные именно этого
  мероприятия (участия, покупки, чеки, присвоенные номера, состояние диалогов) из
  БД (каскад) + привязанный Google-лист. Другие мероприятия не затрагиваются.
  Перед удалением — подтверждение.

### 9.3 Раздел «Чеки / Модерация»
- Очередь покупок с фильтрами по статусу (`manual_review`, `approved`, и т.д.).
- По каждой покупке — **полная детальная информация** (админка показывает всё,
  в отличие от публичной таблицы):
  - **просмотр изображения чека**;
  - привязанный участник: **vk_id** (видимый), **имя в ВК**, **прямая ссылка на
    профиль** (быстрый переход), **имя** (присланное участником), **телефон**;
  - мероприятие, распознанная сумма, кол-во билетов, **присвоенные номера**;
  - статус оплаты (оплачено / не оплачено / на проверке / отклонено).
- Кнопки смены статуса **в обе стороны** (одобрить ↔ отклонить).
- **Отзыв номеров** (`revoke`): если оплата не прошла, чек поддельный или по иной
  причине — админ отзывает уже присвоенные номера. При этом: `poster_numbers`
  покупки удаляются (номера снова свободны и доступны для выдачи другим), строки
  убираются из публичной таблицы, покупка переходит в `revoked`. Действие — с
  подтверждением.
- Возможность вручную скорректировать сумму/кол-во билетов/имя/телефон перед
  одобрением.
- **Частичная/ошибочная оплата** (сумма не кратна цене, меньше цены, чужой
  получатель) — выделяется отдельно и попадает в ручную очередь админу.
- **Поиск** — быстрый поиск участника/чека по данным (имя, имя ВК, vk_id, телефон,
  номер билета, сумма). Должен работать оперативно.

### 9.4 Раздел «Участники»
- Список участников мероприятия с присвоенными номерами.
- Выгрузка/просмотр; соответствует Google-таблице.

### 9.5 Раздел «Розыгрыш / Победители»
- Встроенный генератор: по завершении продаж выбирает **`winners_count` случайных
  номеров** (из настроек мероприятия) **только из оплаченных** (фактически
  присвоенных) номеров — непроданные номера победить не могут. Номера-победители
  не повторяются.
- Показывает победителей (участников с детальной инфой).
- Фиксация результата розыгрыша.

## 10. Google Sheets (зеркало)

- Доступ через service account (Google-аккаунт заказчика/разработчика), `gspread`.
- **Авто-создание** таблицы при создании мероприятия + выдача доступа на чтение
  «всем по ссылке» (публичная таблица участников, как у конкурента).
- **Колонки публичной таблицы — только три: `Номер` | `Имя` | `Оплачено`.**
  Никаких vk_id, телефонов, ссылок — это только в админке (§9.3).
- **Одна строка = один оплаченный номер.** В таблицу попадают **только оплаченные
  (присвоенные) номера**; пустые/непроданные номера НЕ показываются. У участника
  с несколькими номерами — несколько строк (с тем же именем).
- `Имя` — **только имя** участника (присланное им; без фамилии/полного ФИО).
- `Оплачено` — отметка ✅ (в таблице по определению только оплаченное).
- Синхронизация: при присвоении номеров (статус «оплачено») строки добавляются;
  при откате статуса в админке — удаляются из таблицы. БД — источник истины; при
  рассинхроне Sheets перестраивается из БД (кнопка «пересобрать таблицу» в админке).
- Устойчивость к лимитам Google API: запись батчами/с ретраями; падение Sheets
  не блокирует основную логику (БД уже записана).

## 11. Анти-фрод и edge-cases (заложить в реализацию)

- **Дубль чека** — по `receipt_hash` (и/или по распознанным реквизитам/№ операции)
  → ручная очередь с пометкой.
- **Частичная/ошибочная оплата** (сумма не кратна цене / меньше цены / чужой
  получатель) → выделяется и идёт в ручную очередь админу.
- **Оплатил, но не указал имя/телефон** → номера присваиваются (не блокируем),
  участнику уходит напоминание прислать имя и телефон; админ видит пробел и может
  дозаполнить.
- **Отзыв номеров** (поддельный чек / оплата не прошла) → админ отзывает номера,
  они освобождаются и снова доступны для выдачи, строки уходят из публичной таблицы,
  покупка → `revoked`.
- **Исчерпание диапазона номеров** → алерт админу, пауза присвоения.
- **Сообщение без кодового слова / чек без активного мероприятия** → бот молчит.
- **Повторные покупки одним человеком** → накопление номеров (несколько purchases).
- **Разные форматы чека** (фото / скрин / pdf / пересланное сообщение) — обработка
  вложений ВК (`photo`, `doc`).
- **Несколько активных мероприятий одновременно** с разными кодовыми словами —
  поддерживается.
- **Остановка/удаление мероприятия** — остановленное не реагирует; удаление чистит
  только свои данные (каскад), не задевая другие мероприятия.

## 15. Критерии приёмки

1. В админке можно создать мероприятие со всеми полями; Google-таблица создаётся
   и привязывается автоматически.
2. Бот в ответ на кодовое слово шлёт инструкцию с QR.
1. В админке можно создать мероприятие со всеми настройками (цена билета, кол-во
   билетов, кол-во победителей, сроки, шаблоны, QR, получатель, автоподтверждение);
   Google-таблица создаётся и привязывается автоматически.
2. Бот в ответ на кодовое слово активного мероприятия шлёт инструкцию с QR; вне
   сценария и по остановленным мероприятиям — молчит. Несколько мероприятий с
   разными словами работают параллельно.
3. При присылке чека + имени + телефона бот сохраняет чек и данные участника,
   прогоняет OCR (сверка суммы и получателя), ставит статус по правилам §8;
   участнику уходит «чек на проверке».
4. При `auto_confirm=true` и валидном OCR — чек подтверждается автоматически;
   иначе (вкл. частичную/ошибочную оплату) попадает в ручную очередь.
5. Админ видит чек и полную инфо участника (vk_id, ссылка, имя, телефон), меняет
   статус в обе стороны, корректирует сумму/имя/телефон, ищет по участнику/чеку.
6. После подтверждения участнику присваивается `floor(сумма/цена)` уникальных
   случайных номеров (даже без имени/телефона — с напоминанием их прислать), уходит
   2-е сообщение; в публичную Google-таблицу (`Номер | Имя | Оплачено`) добавляются
   строки только по оплаченным номерам.
7. Генератор выбирает `winners_count` случайных номеров из оплаченных и показывает
   победителей с детальной информацией.
8. Вход в админку — 1 аккаунт по логину/паролю из `.env`. Остановка и удаление
   мероприятия работают; удаление чистит только данные этого мероприятия.
9. Всё развёрнуто на сервере, админка доступна по IP, бот работает стабильно.

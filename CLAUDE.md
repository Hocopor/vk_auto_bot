## Протокол сессии (ОБЯЗАТЕЛЬНО)

1. **Старт:** прочитай `STATE.md`, затем нужный раздел `PLAN.md`. НЕ сканируй проект целиком — вся точка входа в этих двух файлах. Если пользователь сказал «продолжай» — продолжай с «Следующего шага» из STATE.md.
2. **Схема работы — оркестратор + сабагенты.** Главная сессия (самая сильная модель) — ОРКЕСТРАТОР: читает состояние, принимает решения, ПРОЕКТИРУЕТ реализацию, ставит задачи, проверяет результат, обновляет STATE/PLAN. Сам код руками почти не пишет — РАБОТУ ДЕЛАЮТ САБАГЕНТЫ с моделью по сложности:
   - `coder-simple` (haiku) — boilerplate, CRUD, шаблоны/разметка, конфиги, манифесты зависимостей, мелкие правки по точному ТЗ;
   - `coder` (sonnet) — основная бизнес-логика, интеграции, тесты;
   - `architect` (opus) — архитектура, сложный дебаг, нетривиальные интеграции, ревью критичного.
   Если кастомные агенты не подхватились — используй обычный сабагент с параметром model (haiku/sonnet/opus). Не используй opus там, где справится haiku. В промпт сабагенту давай ПОЛНОЕ ТЗ: файлы, интерфейсы, ограничения — он стартует с нуля, без контекста сессии.
3. **Оркестратор ПРОЕКТИРУЕТ — сабагент только пишет код.** Главная сессия — самая сильная модель по архитектуре, сабагенты — нет. Поэтому ДО запуска сабагента оркестратор сам полностью продумывает реализацию и записывает её в ТЗ: какие файлы создать/править, какие функции/классы с какими сигнатурами, какие структуры данных и схемы БД, какой алгоритм по шагам, какие библиотеки/API и как именно их применять, какие edge-cases обработать, какие тесты написать. Сабагенту должно остаться ТУПО НАПИСАТЬ КОД по готовому проекту — никакого «придумай сам». Если в ТЗ есть место, где сабагенту пришлось бы принять архитектурное решение — ТЗ не готово, додумай сам и допиши.
4. **Сабагентам в ТЗ всегда запрещать:** любые git-операции (commit/push/reset/checkout) — коммит делает пользователь; правку `STATE.md` и `PLAN.md` — это зона оркестратора.
5. **После КАЖДОГО цикла работы** (завершил блок/задачу — не только в конце сессии!): отметь чекбоксы в `PLAN.md`; обнови в `STATE.md` текущую точку и следующий шаг. Узнал неочевидное (грабли, лимит API, особенность платформы) — сразу допиши в `STATE.md → Нюансы`. Правило: после любого изменения кода STATE.md не должен отставать — ничего не теряем. (Stop-хук `.claude/hooks/check-state.ps1` не даст завершить ответ, если STATE.md отстал от изменений файлов.)
6. **Финиш сессии:** финально обнови `STATE.md` — что сделано и с каким результатом (честно: «тесты прошли» / «застрял на X»), следующий шаг. Важные решения — в `PLAN.md → Журнал решений`.

## Фаза проектирования (когда задача ещё «сырая идея»)

Если проект/фича на стадии абстрактной идеи — НЕ бросаться в код. Сначала превратить идею в конкретику, по шагам:
1. **Уточнение ТЗ.** Прочитать запрос/`TZ.md`, выписать ВСЕ открытые вопросы и развилки (особенно: юридические требования, оплата, интеграции, безопасность, данные пользователей). Задать их пользователю пачкой — не угадывать. Зафиксировать ответы в `TZ.md`/`PLAN.md`.
2. **Архитектура и стек.** Когда требования ясны — выбрать стек и архитектуру (оркестратор/architect), записать решение и его обоснование в `PLAN.md → Журнал решений`.
3. **Декомпозиция в план.** Разбить на этапы и подзадачи с чекбоксами в `PLAN.md`, по каждому — критерий приёмки. Каждая подзадача должна быть на уровне «сабагенту осталось написать код».
4. **Только потом — реализация** по плану, по порядку, циклами (см. протокол сессии).

## О проекте

ВК-бот + веб-админка для розыгрышей «постеров/билетов». Бот реагирует на кодовое
слово, выдаёт инструкцию с QR-кодом оплаты, принимает чек + имя + телефон,
проверяет оплату (OCR + ручная модерация), присваивает участнику случайные
уникальные номера и фиксирует его в публичной Google-таблице. По окончании —
генератор выбирает победителей из оплаченных номеров. Копируем функционал
конкурента (скрины в `tz/`), но не его данные.

- Исходное ТЗ (переписка) — `TZ.md`. **Итоговое ТЗ — `ТЗ_ИТОГ.md`** (главный документ).
- Техническая архитектура — `ARCHITECTURE.md`.
- Ключевое: всё настраивается в админке на каждое мероприятие (цена, кол-во
  билетов, кол-во победителей, сроки, тексты, QR, получатель); несколько
  мероприятий параллельно; остановка и каскадное удаление; отзыв номеров;
  1 админ из `.env`.

## Стек и ограничения

- **Python 3.12.** Бот — `vkbottle` (Long Poll). Админка — FastAPI + Jinja2.
  БД — **PostgreSQL** (SQLAlchemy async + asyncpg, миграции Alembic). OCR —
  **Tesseract** (`pytesseract`, локально, rus+eng). Sheets — `gspread`.
- Хостинг — **cloud.ru** (Linux, РФ — под 152-ФЗ). Админка заказчика — по IP без
  домена. 2 процесса: `vk-bot`, `admin-web` (см. `ARCHITECTURE.md §2`).
- Бюджет/сроки ориентир: ~5000 ₽, ~2 дня. Держать просто.
- Секреты — только в `.env` (VK-токен, DSN, путь к Google SA json, ADMIN_LOGIN,
  ADMIN_PASSWORD_HASH, SESSION_SECRET), в git НЕ коммитить.

## Команды

4. **Оркестратор проектирует — сабагенты пишут код** (см. универсальный протокол выше):

   в `STATE.md → Нюансы`). Бот-флоу без живого ВК — юнит/интеграц. тестами + ручной
   прогон заказчика.
6. **Git и деплой делает АССИСТЕНТ** (заказчик разрешил): `commit` + `push` в GitHub,
   затем деплой С GitHub на сервер. Деплой ТОЛЬКО через git (НЕ SFTP):
   push → `git pull` в `/opt` и `/root` → `alembic upgrade head` → `restart admin-web
   vk-bot` → smoke (`:8080/login=200`). **Инкрементально по группам.** Доступ к серверу —
   `ssh.md` (root@185.228.72.118); коннект с Windows — paramiko `scripts/_ssh_run.py`

7. **Источник задач — `edit.md`** (фидбек заказчика), разложен в `PLAN.md → Фаза 8`
